LLM Wiki 離線工作站：架構與流程

改編自 Karpathy 的 llm-wiki 模式：知識只編譯一次、由 LLM 持續維護， 並在其上建立一條可追溯的硬體設計流程——spec 進來，經 wiki 編譯，產出 Verilog-2001 RTL。 全系統零網路依賴。

01

核心概念：編譯一次，持續維護

為什麼不用 RAG

傳統 RAG 每次查詢都重新檢索原始文件——同樣的綜合工作反覆做、結果不留存。 llm-wiki 反過來：來源進來時就編譯成結構化的 wiki 頁面（Markdown、互相連結、附引用）， 之後查詢直接讀 wiki。知識是持久、複利成長的資產；每加一份來源，整個 wiki 的綜合品質都在提升。

分工明確：人蒐集來源、提問、做裁決；LLM 做所有記帳——更新頁面、補交叉連結、檢查矛盾。 wiki 荒廢的主因是維護太累，而 LLM 不會累、不會忘記更新第 15 個交叉連結。

02

與傳統 RAG 的比較

同一個比喻：RAG 是直譯器（每次執行都重新解讀），llm-wiki 是編譯器（編譯一次、之後直接跑）

傳統 RAG 綜合發生在查詢時——同樣的工作每問一次做一次，結果不留存
建置期 原始文件 → 切塊 chunking → embedding → 向量資料庫
每次查詢 ↻ 每問一次重做 → 相似度檢索 → 撈出零散 chunks → LLM 現場拼裝理解 → 回答 → 丟棄

橘色的三步是重複成本：跨文件的關聯每次都要現場重建， 答得好不好取決於這次剛好撈到哪些 chunks；上次答過的綜合結論下次從零再來。 矛盾的來源各自被撈出來，模型現場硬拼。

LLM Wiki 綜合發生在 ingest 時——編譯一次，之後每次查詢都站在已整理的知識上
ingest 時 ✓ 每份來源只做一次 → 完整閱讀來源 → 整合進 wiki 頁面 → 交叉連結＋標注矛盾
每次查詢 讀 index.md → 讀相關頁面 → 附引用的回答 → 有價值就回存 wiki

重活集中在 ingest 那一次；查詢讀的是已經消化、互相連結、 矛盾已標注的頁面。回答本身還能回存成新頁面——知識不是消耗品，是複利資產。

逐項對照

傳統 RAGLLM Wiki
綜合時機 查詢時，現場拼裝，每次重做 ingest 時做一次，之後直接用
知識留存 無——答完即丟，系統不會變聰明 複利成長——每份來源、每個好答案都沉澱進頁面
檢索單位 零散 chunks（相似度撈出，缺上下文） 完整頁面（人可讀、有結構、有連結）
來源矛盾 矛盾的 chunks 一起被撈出，模型現場硬拼 ingest 時就發現，並列兩說標 ⚠️ 待人裁決
引用追溯 能指到 chunk，難以指到主張層級 每個主張連回來源檔；spec-to-RTL 更是機器可驗證
基礎設施 embedding 模型＋向量資料庫＋檢索管線 Markdown＋git＋SQLite FTS5（純標準函式庫）
離線適配 embedding 模型也要離線部署，管線更重 零額外依賴，天生離線
人的角色 只能調參數（chunk 大小、top-k…） 直接讀寫 wiki——知識庫本身就是給人看的

§

什麼時候 RAG 仍是對的：語料巨大且低價值密度（客服紀錄、全站文件）、 查詢一次性、不需要知識沉澱時，RAG 的「不整理、用時再撈」反而划算。 llm-wiki 適合的是小而精、長期經營、需要追溯的領域——正是離線工作站＋硬體設計的情境。 本 repo 的搜尋層（FTS5）只負責「找到該讀哪頁」，綜合永遠由 wiki 頁面承擔，不是 RAG。

03

四層架構

資訊單向往下流；每一層只依賴上一層

CLAUDE.md
Schema — 規則層 定義 wiki 結構、慣例與全部操作流程，是 LLM 的操作手冊、唯一規範來源。 AGENTS.md（Codex）與 opencode.json（OpenCode）都指回這份，零漂移。
↓ 約束
sources/
原始來源 — 不可變 文章、筆記、spec 文件。inbox/ 待處理 → archive/ 永久保存。 LLM 絕不修改；是所有主張的最終依據。
↓ ingest（編譯）
wiki/
Wiki — LLM 維護的知識層 index.md（目錄）＋ log.md（追加式日誌）＋ pages/ 五分類：concepts · entities · notes · specs（編號需求）· design（設計頁）。
↓ 生成（只讀設計頁）
rtl/ + verif/
產物層 — 可重生 Verilog-2001 RTL、自檢 testbench、filelist、三種商用模擬器 run scripts。 任何一個檔案都能從設計頁重新生成。

目錄一覽

04

三個核心操作

所有 wiki 活動都是這三件事的組合；每次操作 = 一個 git commit

ingest吸收來源

讀 inbox 的新來源，整合進現有頁面（一份來源觸及 10–15 頁是正常的）， 補交叉連結，來源移入 archive，更新 index 與 log。 commit：ingest: 標題

query查詢

先讀 index.md 找頁面（頁數多用 FTS5 搜尋），綜合出每個主張都附引用的答案。 有長期價值的答案回存為 notes 頁面，知識複利。 commit：note: 標題

lint健檢

每累積約 10 次 ingest 執行：抓矛盾、過時主張、孤兒頁、斷鏈、index 失準， 並列出資料缺口請人補來源。 commit：lint: 日期

§

四條鐵律：sources/ 絕不修改 · 每個主張可追溯到來源 · 每次操作後更新 index.md 與 log.md · 來源矛盾時並列兩說標 ⚠️ 待釐清，不擅自裁決。

05

離線工作站的關鍵設計

日常運作零網路；需要網路的事全部推到連網蒐集機

  • 本地推論 — Ollama 跑 Qwen3.6-27B，OpenCode 由 opencode.json 直接載入 CLAUDE.md 為規則並指向本地端點。建議 context ≥ 32K（ingest 要同時裝下來源＋十幾頁相關頁面）。
  • 來源管線 — 連網機完成「網頁→Markdown、圖片本地化、PDF→.md 轉檔」， 整包經 USB 單向帶入 sources/inbox/。離線機備援：PyMuPDF wheel 離線安裝＋ pdf_to_md.py；轉不了的 PDF 列入資料缺口，絕不憑檔名猜內容。
  • 零依賴搜尋 — 分階段：<100 頁直接讀 index；之後用 wiki_search.py（SQLite FTS5 + BM25，純標準函式庫，不需向量庫）。
  • Git 即稽核軌跡 — 每次 LLM 操作一個 commit；框架更新走「整包 repo 資料夾經 USB → 工作站 AI merge」，log.md 設 merge=union 防衝突。本地 wiki 內容永不被更新覆蓋。
06

Spec-to-RTL Flow

wiki 是 spec 與 RTL 之間的編譯層

直接把幾百頁的 spec 丟給本地模型寫 RTL：context 裝不下、無法追溯、spec 改版即災難。 這條 flow 把 spec 編譯成兩級中間表示——先是編號需求，再是模組設計頁——RTL 只從設計頁生成：

sources/ Spec 文件 不可變原文
（v1.0, v1.1…）
→人審
specs/ 編號需求 REQ-UART-001…
ID 永不重編
歧義標 ⚠️
→人審
design/ 設計頁 ports · FSM · 握手
＝RTL 生成契約
→
rtl/ verif/ RTL + TB Verilog-2001
自檢 testbench
→人跑模擬
sim TEST PASSED VCS / Questa / Xcelium
失敗 log 貼回分析
§

核心紀律：RTL 只從設計頁生成，絕不直接讀 spec。 設計頁資訊不足就先回頭補設計頁（必要時補需求頁）。 好處：單次生成的 context 永遠只有一張設計頁的量（27B 模型輕鬆裝下），且每行 RTL 都能追溯。

六個操作

指令做什麼commit
ingest（spec 進 inbox） spec → 需求頁；spec 改版時逐條 diff，變更標 🔶、波及的設計頁自動標 needs-review（影響分析） spec: …
design uart-lite 需求頁 → 架構頁＋每模組一張設計頁；每條需求必須被至少一張設計頁的 implements 承接 design: …
rtl uart_tx 設計頁 → rtl/uart_tx.v，檔頭回鏈設計頁與 REQ ID，更新 filelist rtl: …
verify uart-lite 自檢 TB（結尾必印 TEST PASSED/FAILED）＋三種模擬器 run scripts；模擬由人執行 verify: …
lint 一般健檢＋ trace_check.py：孤兒需求、斷鏈產物、needs-review 未清、矩陣過期 lint: …
flow uart-lite 半自動：design → rtl → verify 一口氣串完（見 08） 各階段照常

機器可驗證的追溯鏈

spec 章節§3.1 ← 需求REQ-UART-004 ← 設計頁 implementsdesign/uart-tx.md ← 檔頭回鏈rtl/uart_tx.v ← tb:verif/tb_uart_lite.v

trace_check.py check 機械化檢查整條鏈（有缺口 exit 1）， trace_check.py matrix 重生追溯矩陣頁。spec 改版時，被波及的範圍一查即知。

內附範例：uart-lite

走完整條 flow 的參考實作 ✓ TEST PASSED · Icarus Verilog 12.0

8N1、16-bit 可程式 baud divisor、valid/ready 握手。從 spec 文件出發： 13 條需求 → 3 張設計頁（架構＋TX＋RX）→ 3 個 RTL 模組 → loopback 自檢 TB （4 bytes 背靠背傳輸＋壞停止位元的 frame error 測項），追溯零缺口。 wiki/log.md 記錄了每個階段的長相，是日後真實專案的範本。

08

半自動 flow 與大型 design

flow <block> — 串接但不失控

需求單純時一個指令串完 design → rtl → verify。半自動與全自動的分界線劃在四個地方：

  • Spec-Ingest 永遠不在 flow 裡 — spec 解讀必須人審，flow 從審過的需求頁起跑。
  • 啟動 gate — 需求頁有未清的 ⚠️ / 🔶 就拒絕啟動。歧義不許自動流過。
  • 每階段照常各自 commit — git 歷史與手動逐步完全相同；卡住就停在該階段報告，已完成的不回滾。
  • 模擬仍由人執行 — flow 的產出是「等待模擬的 RTL + TB」，不是「已驗證的設計」。

大型 block（如 MIPI CSI-2）的分解規則

模組數 > ~5 或需求數 > ~30 時強制加四條：

需求分域

spec 按功能域拆多張需求頁，各用獨立前綴： REQ-DPHY-*、REQ-PKT-*、REQ-PIXEL-*

介面定義頁

模組間介面集中在 design/<block>-interfaces.md，唯一權威； 設計頁只引用介面名，消滅兩端認知不一致

遞迴架構頁

頂層只分解到子系統，子系統各有架構頁再分到模組； 每張架構頁直接子節點 ≤ ~7

分層 verify

模組 TB（含錯誤注入）→ 子系統 TB → top 整合 TB； flow 一次只跑一個子系統

不管 design 多大，單次 RTL 生成讀的都是固定三小份：模組設計頁＋所屬架構頁接線章節＋用到的介面定義。 context 有界——這是 27B 本地模型能做 CSI-2 量級的關鍵。
